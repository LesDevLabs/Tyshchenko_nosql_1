# Tyshchenko_nosql_1



1. Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?

Залежності від задачі, іноді легше логічне обєднання, що дозволяє легше опрацьовувати дані одного типу. Залишаючи дані плоско, легше отримати конкретнє значення, але втрачається ідея однакової належності, сенсу даних.

2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?

Однакові за сенсом використання дані варто зберігати разом, масив спеціально придуманий для зберігання множини однотипних значень. Він дозволяє шукати точно по значенню, фільтрувати треки конкретного виконавця або знаходити колаборації. 

3. Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?

$out повністю замінює колекцію результатом пайплайну, видаляється повністю і створюється заново. 
$merge оновлює і\чи обєднує результат з існуючою колекцією не чіпаючи решту даних які вже були в документі.


---

1. Для чого використовується інструкція $unwind?

$unwind - розгортає масив у окремі документи. Без цього, як приклад, $group в даній логіці бдуе групувати по 1 масиву, а не по кожному артисту окремо.

2. Чим $stdDevPop відрізняється від $stdDevSamp?

$stdDevPop рахує наскількі значення розкидані від середьного, для всіх наявних даних.
$stdDevSamp працює з вибіркою, коли маєш лише частину даних, але хочеш зрозуміти розкид у всьому наборі. ( як екзит пол )

---

1. У запиті 1 ми фільтруємо виконавців, у яких менше 5 треків. Як зміниться результат, якщо знизити поріг до 1? А що станеться, якщо вибирати виконавців із більш ніж 50 треками? Поясніть результат.

1 пісня - статистики немає, бо 1 це не вибірка. 50 - дуже мало артистів попаде у вибірку. 
Треба правильно підібрати значення для опрацювання наявних даних, для кращого результату.
Якщо береться підходяща вибірка по популярності, а не просто популярність, то треба знайти межу з якої артист стає артистом, а не власником 1 пісні. 


2. У запиті 3 ми фільтруємо жанри з менше ніж 100 треками. Чи зміниться результат, якщо знизити поріг до 50? Поясніть результат.

Якщо неправильно підібрати параметр, до результат буде не надійним, і матиме вищу можливість похибки. В цьому випадку 50 - не виглядає як недостатнім числом, враховуючи кількість даних. 
Жанри з малою вибіркою можуть спотворити результат.

Питання: Чи є цей запит покривним (covered query)? Надайте розгорнуту та обґрунтовану відповідь у файлі README.

Без обовязкового _id - не є. Запит має все в індексі, щоб повернути дані, без читання документа, окрім _id, який повинен бути за замовчуванням.

Тому буде читати документ і запит не буде covered query, якщо додати _id: 0 ( щоб не повертав ), то можна отримати covered query для поточного запиту.
